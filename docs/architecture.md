# Архитектура Vodomerka

## Цели

- Renderer не знает деталей Electron IPC и работает через единый типизированный контракт.
- Код приложения зависит только вниз: `app → components → shared`.
- Общий код не импортирует UI-композицию или Electron main process.
- Изменения архитектуры делаются инкрементально, без одновременного переписывания работающих экранов.

## Слои

### `contracts`

Общие типы и имена IPC-каналов. Это единственная разрешённая точка связи между Electron main/preload и renderer. Слой не содержит бизнес-логики и не зависит от `src` или `electron`.

### `electron`

Main process, preload, SQLite и IPC adapters. Он отвечает за системные возможности, хранение и безопасность границы Electron, но не за React-состояние и отображение.

### `src/app`

Composition root renderer:

- `providers` — порядок и подключение глобальных провайдеров;
- `navigation` — состояние навигации верхнего уровня;
- `shell` — главное и дополнительное окна;
- `App.tsx` — выбор shell и запуск приложения.

### `src/components`

Работающие экраны и крупные UI-блоки. Это переходный слой: новые большие возможности следует создавать в `src/features`, а переиспользуемые доменные представления — в `src/entities`.

### `src/shared`

Переиспользуемые API-клиенты, доменная модель, platform adapters, настройки, hooks и UI primitives. Этот слой не может импортировать `app` или `components`.

## Направление зависимостей

```text
src/app
  ↓
src/features (для новых возможностей)
  ↓
src/entities (для новых доменных модулей)
  ↓
src/components (существующий UI)
  ↓
src/shared

electron → contracts ← renderer
```

`features` и `entities` вводятся по мере развития, а не пустыми директориями. Существующие компоненты мигрируют туда только вместе с реальной доработкой соответствующей области.

## Обязательные проверки

Перед завершением изменения:

```bash
npm run check
```

Команда проверяет границы слоёв, TypeScript для renderer и Electron, затем production-сборку без упаковки DMG. Полная упаковка выполняется через `npm run build`.

## Правила развития

1. Новый IPC-канал сначала добавляется в `contracts/ipc.ts`, затем реализуется в main process и preload.
2. Компоненты не обращаются к `ipcRenderer`, SQLite или `fetch` напрямую.
3. Side effects и загрузка данных выносятся из представления в hook/model или repository.
4. Доменная логика остаётся чистой и не импортирует React.
5. Публичные импорты модуля оформляются через `index.ts`, когда у модуля появляется больше одного внешнего потребителя.
6. Кэш всегда принадлежит repository/store, имеет явную стратегию invalidation и не дублируется в компонентах.
7. Большие миграции выполняются вертикальными срезами: контракт → data/model → UI → проверки.
